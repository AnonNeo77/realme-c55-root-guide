# 07 - Flashing the Patched Boot Image

## Overview

Flashing the patched boot image on the Realme C55 (RMX3710) differs from most
Android devices. The standard bootloader Fastboot interface is **not exposed**
on retail firmware. The available flashing path depends entirely on the
firmware version installed on the device.

---

## The Firmware Problem

Realme locked down MTKClient-accessible interfaces in later firmware builds:

| Android Version | Bootloader Interface | MTKClient Access |
|-----------------|----------------------|------------------|
| Android 13      | Not exposed  | Available        |
| Android 14 (<1800 build) | Not exposed | Available     |
| Android 14 (≥1800 build) | Not exposed | Patched         |
| Android 15      | Not exposed          | Patched          |

If the device is running Android 14 (build ≥1800) or Android 15, the
BootROM/Preloader interface that MTKClient relies on has been patched. The
exploit path no longer works on those builds.

---

## Downgrading to a Compatible Firmware

To reach a firmware version where MTKClient is operational, the device must be
downgraded using a Realme downgrade OTA package.

**Downgrade path:**

Downgrade packages are distributed by the community (e.g., on XDA and Realme
community forums). These are official Realme OTA packages repurposed for
downgrading, not custom firmware.

> **Data loss:** The downgrade process performs a factory reset. Back up all
> data before proceeding.

After downgrading to Android 13, the BootROM/Preloader interface is accessible
via MTKClient.

---

## Prerequisites Before Flashing

Before replacing the boot image, confirm the following:

- Device is on a compatible firmware (Android 13 or Android 14 pre-1800 build)
- Stock `boot.img` extracted from the **exact** installed firmware build
- Patched `boot.img` generated by Magisk from that stock image
- Original stock `boot.img` backed up separately
- Battery above 60%

Using a patched image built from a mismatched firmware version will likely
result in a boot failure or AVB verification error.

---

## Bootloader Unlocking via MTKClient

The bootloader unlock is performed through the MediaTek Preloader interface
using [MTKClient](https://github.com/bkerler/mtkclient).

MTKClient communicates with the device at the BootROM/Preloader stage — before
the main OS loads — which is what allows operations that the locked bootloader
would otherwise block.

```bash
python mtk da seccfg unlock
```

After the unlock command completes, the device reboots. The bootloader is now
in an unlocked state. Android Verified Boot will display a warning on every
subsequent boot — this is expected behavior.

> **Note:** Bootloader unlock triggers a factory reset on first boot. This is
> enforced by the bootloader itself, not by Realme software.

---

## Flashing the Patched Boot Image

With the bootloader unlocked, the patched `boot.img` is written to the boot
partition using MTKClient.

```bash
python mtk w boot patched_boot.img
```

MTKClient handles the partition write through the Preloader interface.
Fastboot is not required or used at any point in this process.

---

## First Boot

The first boot after flashing takes longer than usual. The sequence is:

1. Bootloader loads the patched `boot.img`
2. Linux kernel starts
3. Modified ramdisk initializes Magisk
4. Android continues the normal boot sequence
5. Magisk completes initialization during early userspace startup (`init` stage)

If the device boots into Android normally, the patched image is functioning.

---

## Verifying Root Access

After Android finishes booting:

1. Open the **Magisk** app
2. Confirm Magisk shows as **Installed** (not requiring additional setup)
3. Verify the installed version matches the version used to patch the image
4. Test a root-dependent app (e.g., a root checker or `adb shell su`) to
   confirm the `su` binary is accessible

---

## Common Failure Modes

| Symptom | Likely Cause | Fix |
|---------|--------------|-----|
| Boot loop | Mismatched boot image firmware | Flash original stock `boot.img` |
| Stuck at boot logo | Corrupted patch or write error | Re-flash via MTKClient |
| AVB warning on every boot | Expected post-unlock behavior | No action needed |
| Magisk not initialized | Patch applied to wrong boot.img | Repatch from correct stock image |
| MTKClient fails to connect | Incompatible firmware build | Downgrade to Android 13 |

---

## Recovery

If the device does not boot after flashing, restore the stock `boot.img`:

```bash
python mtk w boot stock_boot.img
```

This does not re-lock the bootloader. The device will boot normally on the
original image.

---

## Summary

The Realme C55 rooting workflow is constrained by firmware-level changes that
patched MediaTek's Preloader interface in Android 14 (build ≥1800) and
Android 15. Reaching a working state requires downgrading to Android 13 via
community-distributed Realme OTA packages, unlocking the bootloader through
MTKClient, and flashing the patched boot image through the same interface.

Fastbootd is present on the device but does not provide bootloader unlock
functionality and is not part of this workflow.
